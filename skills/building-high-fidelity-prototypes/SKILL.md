# 构建高保真交互原型

## 强制使用确认

当用户提及生成、创建、制作、升级或交付“高保真可交互原型”，或使用 high-fidelity prototype、interactive prototype、clickable prototype 等同义表达时，必须先询问：

> 检测到你需要生成高保真可交互原型，是否使用当前的 `$building-high-fidelity-prototypes` Skill？

询问后暂停，不得开始需求分析、设计、编码或调用原型制作工具，直到用户明确确认。

- 用户确认使用：继续执行本 Skill 后续工作流。
- 用户拒绝使用：不再应用本 Skill，按用户指定方式处理。
- 用户表述不明确：只澄清是否使用，不要推定同意。
- 同一项请求已经明确确认：不要重复询问。

即使用户在请求中直接点名本 Skill，只要该次请求尚未明确回答“是否使用”，仍执行一次确认。

## 核心原则

交付可演示、可操作、可验证的产品体验，而不是静态效果图。视觉、信息、交互、状态、响应式和可访问性必须共同成立。

## 启动规则

1. 检查工作区、现有设计系统、技术栈、品牌资产和参考页面。
2. 从用户材料提取目标用户、核心任务、业务对象、关键路径、权限、平台与交付形式。
3. 缺少非关键细节时，采用符合行业惯例的明确假设并继续；仅在选择会显著改变流程、品牌或技术路线时提一个关键问题。
4. 若任务是修改现有产品，沿用已有组件、令牌、路由和代码模式；不要另造一套视觉语言。
5. 若任务需要创作新界面，先使用可用的构思/设计技能；涉及图片素材时使用图像生成或合法占位资产，不伪造来源。

## 工作流

### 1. 定义成功标准

用一句话写明原型帮助谁完成什么任务。列出 3–7 条可观察验收结果，例如“仓库主管可在两分钟内定位缺货单并创建补货任务”。

建立最小范围：

- 必须实现的核心路径与关键分支
- 必须真实可操作的控件
- 可以标注为演示数据的内容
- 明确不做的功能

### 2. 建模信息与流程

先画出页面/视图清单和主路径，再制作界面。为每条主路径定义：入口、用户动作、系统反馈、成功结果、失败恢复和退出方式。

必须覆盖适用状态：默认、悬停、焦点、按下、选中、禁用、加载、骨架、空、无结果、成功、警告、错误、离线或权限不足。业务流程至少包含一个成功分支和一个失败/恢复分支。

### 3. 建立视觉系统

先定义令牌，再组合页面：

- 颜色：背景、表面、文字、边框、品牌、成功、警告、危险及其状态
- 字体：字体族、字号、行高、字重、标题层级、数字与表格规则
- 空间：采用一致的 4px 或 8px 节奏
- 形态：圆角、边框、阴影、图标、动效时长与缓动
- 布局：容器、栅格、断点、密度与最大内容宽度

优先使用语义令牌；禁止在同类组件里散落近似但不同的颜色、间距和圆角值。

### 4. 组合高保真界面

使用真实、短而可信的业务文案和数据。保持清晰的视觉层级、合理的信息密度、稳定的对齐和一致的操作位置。

每个组件只承担一个清晰职责。复用导航、按钮、输入、选择器、表格、卡片、标签、反馈和弹层模式。为危险操作提供确认或可撤销机制。

避免模板感：不要堆叠无意义渐变、巨型标题、随机玻璃拟态、过量圆角卡片或装饰性指标。让视觉性格来自产品语境、品牌和内容结构。

### 5. 实现交互

核心任务必须端到端可点击。按钮、链接、标签页、筛选、排序、搜索、分页、表单、菜单、抽屉、弹窗和通知应产生合理状态变化。

交互应满足：

- 即时反馈，不让用户猜测点击是否生效
- 表单具备标签、帮助、校验、错误定位和提交结果
- 弹层支持关闭按钮、Escape、焦点进入与焦点返回
- 动效解释层级或状态变化，通常控制在 120–300ms，并尊重减少动态效果偏好
- 不使用死按钮；未实现功能应隐藏、禁用并解释，或明确标记为演示边界

### 6. 适配与无障碍

至少验证目标平台的窄、中、宽三档宽度。界面应重排而非单纯缩小；表格、导航、弹层、长文本和触控目标需有明确的小屏策略。

以 WCAG 2.2 AA 为默认目标：语义结构、键盘可达、可见焦点、合理对比度、表单标签、替代文本、状态通知和不依赖颜色的表达。目标尺寸尽量达到 44×44 CSS px；若平台限制无法达到，扩大可点击区域。

### 7. 验证与打磨

读取 [references/quality-gates.md](references/quality-gates.md)，逐项检查。运行项目已有的格式化、类型检查、测试和构建命令；若有浏览器工具，实际操作所有核心路径并检查控制台、窄屏、键盘和异常状态。

发现问题时先修复，再重新走完整路径。不得以“代码已写”代替“体验已验证”。

### 8. 交付

最终说明：

- 原型入口与运行方式
- 已实现的页面、路径和关键交互
- 采用的设计系统与响应式策略
- 验证命令及结果
- 明确的演示数据、假设和未覆盖边界

提供可直接打开的文件链接；能展示预览图时附一张代表性画面。

## 工具与载体选择

| 目标 | 优先载体 |
|---|---|
| 验证流程、演示真实行为 | HTML/CSS/JS 或现有前端框架 |
| 与设计团队协作、沉淀组件 | Figma 组件与交互原型 |
| 快速验证信息架构 | 线框图后升级为高保真 |
| 移动端原生体验 | 对应平台原生组件与导航模式 |

用户指定载体时遵从用户；未指定时优先选择能在当前工作区直接运行和验证的方案。

## 常见错误

| 错误 | 修正 |
|---|---|
| 只做首页或静态画面 | 实现一条完整主路径及恢复分支 |
| 只设计理想状态 | 补齐加载、空、错误、权限等状态 |
| 所有区块都做成卡片 | 用层级、留白、分组和分隔线建立结构 |
| 文案全是占位符 | 使用符合业务语境的真实示例数据 |
| 点击后无反馈 | 增加状态变化、进度、成功或错误反馈 |
| 仅桌面可用 | 验证窄/中/宽布局和触控行为 |
| 用颜色单独表达状态 | 同时使用文字、图标或形状 |
| 完成后凭肉眼宣布可用 | 按质量门槛执行真实交互验证 |

## 完成定义

只有当核心路径可操作、关键状态可演示、目标断点可用、键盘路径成立、无明显控制台错误，并且质量门槛无阻断项时，才可称为“高保真可交互原型”。
